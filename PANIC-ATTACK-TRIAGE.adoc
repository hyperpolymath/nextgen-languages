**High UnsafeCode cluster** (24+ findings in Rust/Zig FFI layers,
Lean 4/Idris 2 proof tactics, JavaScript `eval`-equivalent constructs): these
need per-file review. Each unsafe block either has a legitimate justification
(and should get a `// SAFETY:` comment), or is an accidental use of a banned
pattern (and should be removed). Budget: 30 minutes per block, ~12 hours
total. **Cannot complete in this session.** Recommended: schedule per-submodule
security review sessions, one language at a time.

**DynamicCodeExecution (4 findings):** JavaScript `eval()` or `new Function()`

need eval; search UIs often do not. **Review required** — likely some are
legitimate and some are replaceable.

== Standing recommendation

**panic-attack should become part of the per-repo `Mustfile` pre-commit gate
once each submodule has been swept clean once.** After each language
completes its initial panic-attack sweep (one dedicated session per language),
the pre-commit gate prevents regressions. Currently the sweeps have not
happened, so turning on the gate would block every commit on pre-existing
findings.

**Signal-to-noise:** panic-attack's `PanicPath` category has too many
false positives in test files and intentional invariant panics. Recommendation
for a panic-attack enhancement: allow `# panic-attack: allow` inline comments
on individual lines, or file-level `// panic-attack: test-file` annotations
that exclude known-safe files from the PanicPath count.
